## Κώδικας Συμπεριφοράς

Με τη συμμετοχή σας σε αυτό το έργο, αναμένεται να τηρείτε έναν επαγγελματικό και σεβαστικό τόνο. Παρακαλούμε:

- Να είστε ευγενικοί και σεβαστικοί προς όλους τους συνεισφέροντες
- Να δέχεστε εποικοδομητική κριτική
- Να εστιάζετε σε αυτό που είναι καλύτερο για την κοινότητα
- Να δείχνετε ενσυναίσθηση προς τα άλλα μέλη της κοινότητας

## Πώς να Συνεισφέρετε

### Βήματα για Συνεισφορά

1. **Κάντε Fork το αποθετήριο**: Κάντε κλικ στο κουμπί "Fork" στο επάνω δεξί μέρος της σελίδας
2. **Κλωνοποιήστε το fork σας**:
   ```bash
   git clone https://github.com/your-username/testing.git
   cd testing
   ```
3. **Δημιουργήστε ένα νέο κλαδί**:
   ```bash
   git checkout -b my-feature-branch
   ```
4. **Κάντε τις αλλαγές σας**: Τροποποιήστε τα αρχεία όπως χρειάζεται
5. **Δοκιμάστε τις αλλαγές σας**: Βεβαιωθείτε ότι όλες οι δοκιμές περνούν
6. **Κάντε commit τις αλλαγές σας**:
   ```bash
   git add .
   git commit -m "Περιγραφική περιγραφή των αλλαγών σας"
   ```
7. **Ανεβάστε στο fork σας**:
   ```bash
   git push origin my-feature-branch
   ```
8. **Ανοίξτε ένα Pull Request**: Πηγαίνετε στο αρχικό αποθετήριο και κάντε κλικ στο "New Pull Request"

## Αναφορά Σφαλμάτων

Εάν βρείτε ένα σφάλμα, παρακαλούμε δημιουργήστε ένα issue με τις ακόλουθες πληροφορίες:

- **Τίτλος**: Σύντομη και περιγραφική περίληψη του προβλήματος
- **Περιγραφή**: Λεπτομερής εξήγηση του σφάλματος
- **Βήματα για την Αναπαραγωγή**: Βήμα προς βήμα οδηγίες για την αναπαραγωγή του σφάλματος
- **Αναμενόμενη Συμπεριφορά**: Τι περιμένατε να συμβεί
- **Πραγματική Συμπεριφορά**: Τι συνέβη πραγματικά
- **Περιβάλλον**: Έκδοση Ruby, λειτουργικό σύστημα, κ.λπ.

## Πρόταση Νέων Λειτουργιών

Εάν έχετε μια ιδέα για μια νέα λειτουργία:

1. Ελέγξτε πρώτα εάν υπάρχει ήδη ένα issue για αυτήν τη λειτουργία
2. Εάν όχι, δημιουργήστε ένα νέο issue που περιγράφει:
   - Τη λειτουργία που προτείνετε
   - Γιατί θα ήταν χρήσιμη
   - Πώς θα λειτουργούσε ιδανικά

## Διαδικασία Pull Request

Παρακαλούμε ακολουθήστε αυτά τα βήματα για τα Pull Requests σας:

1. **Ενημερώστε την τεκμηρίωση**: Εάν οι αλλαγές σας επηρεάζουν τη χρήση του έργου
2. **Προσθέστε δοκιμές**: Για κάθε νέα λειτουργία ή διόρθωση σφάλματος
3. **Βεβαιωθείτε ότι όλες οι δοκιμές περνούν**: Εκτελέστε `ruby alpaca_test.rb`
4. **Ακολουθήστε το στυλ κώδικα**: Δείτε παρακάτω για λεπτομέρειες
5. **Γράψτε σαφή μηνύματα commit**: Χρησιμοποιήστε παρόντα χρόνο ("Προσθήκη λειτουργίας" όχι "Προστέθηκε λειτουργία")
6. **Ενημερώστε το CHANGELOG**: Εάν υπάρχει

### Κριτήρια Αποδοχής Pull Request

Τα Pull Requests θα γίνονται merged όταν:

- Όλες οι δοκιμές περνούν
- Ο κώδικας ακολουθεί τις οδηγίες στυλ
- Η τεκμηρίωση είναι ενημερωμένη
- Υπάρχει τουλάχιστον μία έγκριση από έναν συντηρητή

## Οδηγίες Στυλ Κώδικα

Παρακαλούμε ακολουθήστε αυτές τις οδηγίες κατά τη συγγραφή κώδικα:

### Ruby

- Χρησιμοποιήστε 2 κενά για εσοχές (όχι tabs)
- Διατηρήστε τις γραμμές κάτω από 80 χαρακτήρες όπου είναι δυνατόν
- Χρησιμοποιήστε snake_case για μεταβλητές και μεθόδους
- Χρησιμοποιήστε CamelCase για κλάσεις και modules
- Προσθέστε σχόλια για περίπλοκη λογική
- Αποφύγετε την αφήνοντας `puts` ή `binding.pry` στον κώδικα παραγωγής

### Παράδειγμα

```ruby
class Alpaca
  attr_accessor :name, :age

  def initialize(name, age)
    @name = name
    @age = age
  end

  def display_info
    "Name: #{@name}, Age: #{@age}"
  end
end
```

## Απαιτήσεις Δοκιμών

- Όλα τα νέα χαρακτηριστικά πρέπει να συνοδεύονται από δοκιμές
- Οι διορθώσεις σφαλμάτων πρέπει να περιλαμβάνουν δοκιμές που επαληθεύουν τη διόρθωση
- Εκτελέστε τις δοκιμές με: `ruby alpaca_test.rb`
- Βεβαιωθείτε ότι όλες οι δοκιμές περνούν πριν υποβάλετε ένα Pull Request

### Εγγραφή Δοκιμών

Οι δοκιμές θα πρέπει να:
- Είναι σαφείς και περιγραφικές
- Να δοκιμάζουν μία λειτουργία τη φορά
- Να περιλαμβάνουν τόσο θετικές όσο και αρνητικές περιπτώσεις
- Να είναι ανεξάρτητες και να μπορούν να εκτελούνται με οποιαδήποτε σειρά
